| 项目 | 说明 |
|------|------|
| 文档版本 | v1.1 |

| 目标用户规模 | 约 100 人 |

| 安全要求 | 端到端加密（E2EE） |

---

## 1. 项目概述

### 1.1 背景

为公司内部约 100 名用户提供轻量级即时通讯能力，支持 Android、iOS 移动端，服务器部署在境外云平台并通过公网访问。通讯内容采用端到端加密，服务器无法读取消息明文。

### 1.2 目标

- 实现点对点聊天、群聊、图片与文件收发
- 个人资料仅包含用户名与头像
- 同一账号多设备数据一致
- 群聊及群公告均不提供历史记录给新加入成员
- 功能从简，不追求企业级 IM 完整度

### 1.3 非目标（第一期不做）

- 语音/视频通话
- 已读回执、消息撤回（可选后续迭代）
- 复杂组织架构、部门树、SSO
- 服务端可读的消息审计与全文检索
- 内网/VPN 专属部署
- 桌面聊天客户端（PC 仅用于运维，可选简易管理页）

## 4. 安全需求

### 4.1 传输安全

| 编号 | 需求 |
|------|------|
| SEC-T01 | 全站 HTTPS，禁用明文 HTTP（80 仅跳转 443） |
| SEC-T02 | WebSocket 仅使用 WSS |
| SEC-T03 | TLS 1.2 及以上，建议启用 HSTS |
| SEC-T04 | 可选：Cloudflare 前置，隐藏源站 IP、基础 DDoS 防护 |

### 4.2 端到端加密

| 编号 | 需求 |
|------|------|
| SEC-E01 | 消息正文、群公告、文件内容均在客户端加密后再上传 |
| SEC-E02 | 服务端不得持有消息明文或群对称密钥明文 |
| SEC-E03 | 单聊：采用 X3DH / Double Ratchet 或等效安全会话协商（实现阶段选型） |
| SEC-E04 | 群聊：采用 Epoch + 群消息密钥（GMK），成员变更时轮换 |
| SEC-E05 | 用户身份密钥对（Identity Key）长期保存于客户端安全存储 |
| SEC-E06 | 多设备同步包使用用户主密钥（UMK）加密后经服务器中继 |

### 4.3 应用安全

| 编号 | 需求 |
|------|------|
| SEC-A01 | 所有 REST API 与 WSS 连接必须校验 JWT |
| SEC-A02 | 未授权用户不得访问非所属会话的消息与文件 |
| SEC-A03 | 登录接口限流，防暴力破解 |
| SEC-A04 | JWT_SECRET、数据库密码等仅存服务器环境变量，禁止入库代码仓库 |
| SEC-A05 | 日志不得记录密码、完整 Token、消息明文 |
| SEC-A06 | 可选：移动端 Certificate Pinning |

### 4.4 密钥丢失策略

| 编号 | 需求 |
|------|------|
| SEC-K01 | 用户丢失全部设备且未做加密云备份时，历史消息不可恢复（产品需明确告知） |
| SEC-K02 | 管理员无法代用户解密消息（E2EE 设计约束） |

## 5. 加密与密钥设计

### 5.1 密钥层次

```
用户层
├── Identity Key Pair (IK)      身份密钥，注册时生成
├── User Master Key (UMK)       加密多设备同步包
└── Signed PreKey (可选)        单聊密钥协商

单聊层
└── Session Keys                双方 ratchet 派生，多设备间经 device_sync 同步状态

群聊层
├── Epoch (整数，服务端可见)     成员变更时递增
└── GMK_epoch (群消息密钥)      仅加密该 epoch 内的所有群消息（含 announcement 类型）
```

### 5.2 群聊不可见历史 — 规则说明

1. 建群时 epoch = 0，生成 `GMK_0`，分发给初始成员。
2. 新成员加入时 epoch 递增，生成 `GMK_n`，**仅分发给当前在群成员**（含新成员）。
3. 新成员 **不接收、不存储** 任何旧 epoch 的 GMK。
4. 入群前产生的群消息与群公告（均为 `type=announcement` 或普通消息）均用旧 GMK 加密，新成员 **密码学上无法解密**。
5. 老成员保留已收到的旧 GMK，可继续查看本地已缓存的历史。
6. 成员退群时 epoch 递增，退群者不再接收新 GMK，无法解密后续消息。

### 5.3 群公告与群聊关系统一

| 项目 | 说明 |
|------|------|
| 加密密钥 | 与普通群消息相同，均使用当前 epoch 的 GMK |
| 历史可见性 | 与群聊一致，新成员不可见入群前公告 |
| 差异 | 仅 UI 展示与 `message.type` 字段不同（如置顶样式、公告标签） |
| 不采用 | 独立的群公告密钥（GAK） |

### 5.4 多设备密钥同步

**方式一：链接设备（推荐）**

1. 已登录设备展示二维码（含一次性 token）。
2. 新设备扫码，经服务器 TLS 通道中继 UMK 加密的同步包。
3. 同步包含：身份私钥材料、单聊会话状态、各群当前 GMK 及必要 epoch 密钥、device_id 登记信息。
4. 新设备写入安全存储后，拉取服务器密文并解密，与旧设备一致。

**方式二：加密云备份**

1. 用户设置备份口令（不上传服务端）。
2. 客户端将密钥材料用 PBKDF2 + AES-GCM 加密后上传 `encrypted_key_backup`。
3. 新设备登录后输入口令下载解密；忘记口令则历史不可恢复。

**实时同步**

- 用户所有设备共享同一套密钥材料。
- 收到新 epoch 的 GMK 时，主设备经 `device_sync` 消息同步至同账号其它在线设备。
- 服务端按 `user_id` 向所有 WebSocket 连接推送新消息密文。

## 7. 数据模型

### 7.1 服务端

```sql
-- 用户
users (
  id, email, username, password_hash, avatar_url,
  identity_public_key,  -- Base64
  email_verified_at,    -- 可选：邮箱验证时间
  created_at, updated_at
)
-- email UNIQUE, username UNIQUE

-- 密码重置令牌
password_reset_tokens (
  id, user_id, token_hash,   -- 仅存哈希，防泄露
  expires_at, used_at, created_at
)

-- 设备
user_devices (
  id, user_id, device_id, device_name,
  refresh_token_hash, push_token, platform,
  last_active_at, created_at
)

-- 加密密钥备份（可选）
encrypted_key_backups (
  user_id, ciphertext_blob, version, updated_at
)

-- 会话
conversations (
  id, type,           -- 'private' | 'group'
  name,               -- 群名称，单聊可为空
  epoch,              -- 群当前 epoch，单聊忽略
  owner_id,           -- 群主
  created_at
)

conversation_members (
  conversation_id, user_id, joined_at, left_at
)

-- 消息（均为密文）
messages (
  id, conversation_id, sender_id,
  type,               -- text | image | file | announcement
  ciphertext,
  epoch,              -- 群消息必填
  file_id,
  created_at
)

-- 文件（密文）
files (
  id, uploader_id, storage_path,
  mime, size_bytes, created_at
)
```

### 7.2 客户端本地（SQLite）

- 解密后的消息缓存（可选，便于列表展示）
- 单聊 session 状态
- 群 `GMK` 按 `(group_id, epoch)` 存储
- 同步游标 `last_message_id`

| POST | /api/device-link/complete | 完成链接（新设备） |
| GET | /api/health | 健康检查 |

### 8.2 WebSocket

**连接：** `wss://im.example.com/ws?token=<jwt>`

| 方向 | 事件 | 说明 |
|------|------|------|
| C→S | join | 订阅会话 `{ conversation_id }` |
| C→S | send | 发送密文消息 |
| C→S | key_relay | 密钥分发 / device_sync 中继 |
| C→S | gmk_request | 成员向服务端请求，由服务端转发群主补发 GMK |
| S→C | message | 新消息密文 |
| S→C | key_relay | 密钥相关密文包 |
| S→C | gmk_request | 转发给群主的补钥请求（含 requester、epochs） |
| S→C | epoch_updated | 群 epoch 变更通知（不含 GMK 明文） |

## 11. 关键业务流程

### 11.1 新用户注册

1. 客户端生成 IK，上传公钥。
2. 填写邮箱、用户名、密码完成注册。
3. 使用邮箱 + 密码登录，建立 WSS，进入会话列表。

### 11.2 邮箱找回密码

1. 用户在「忘记密码」页输入注册邮箱。
2. 服务端生成随机 token（有效期如 30 分钟），哈希存库，经 SMTP 发送重置链接：  
   `https://im.example.com/reset-password?token=xxx`（App 可用 Deep Link 或用户在 App 内输入邮件中的验证码）。
3. 用户打开链接/输入 token + 新密码，服务端校验未过期且未使用后更新 `password_hash`。
4. 作废该用户全部 `refresh_token`，已登录设备需重新登录。
5. **E2EE 说明**：若用户无设备链接且无加密云备份，重置密码后仍无法解密旧消息；UI 需明确提示。

### 11.3 单聊发消息

1. 与对方的 session key 加密明文。
2. WSS 发送密文，服务端存库并推送给对方所有在线设备。
3. 发送者其它设备同用户连接亦收到推送。

### 11.4 建群并发消息

1. 群主创建群，epoch=0，生成 GMK_0，向初始成员发送 welcome 密文包。
2. 成员发群消息或群公告：均用当前 GMK 加密，`type` 区分展示。
3. 服务端广播密文给在线成员。

### 11.5 新成员入群

1. 群主邀请 → epoch 递增 → 生成新 GMK。
2. 向现有成员分发新 GMK 密文包；向新成员发送仅含新 GMK 的 welcome 包。
3. 新成员只能解密入群后的消息与公告；**看不到任何历史**。

### 11.6 新设备链接

1. 旧设备打开「链接设备」展示二维码。
2. 新设备扫码，接收 UMK 加密同步包。
3. 新设备拉取各会话密文，本地解密，与旧设备一致。

## 12. 验收标准（第一期）

- [ ] 两用户单聊可收发文本、图片、文件，抓包与服务端库中均为密文
- [ ] 群聊可发普通消息与群公告，新成员入群前消息与公告均不可见
- [ ] 老成员退群再入群，按新 epoch 规则不可见中间段消息（若产品定义为全新 epoch）
- [ ] 同一用户两台设备链接后，会话与消息内容一致
- [ ] 未完成链接的新设备无法解密历史
- [ ] 全站 HTTPS/WSS，未登录无法访问消息接口
- [ ] 文件下载需鉴权且非会话成员无法下载
- [ ] 邮箱注册、登录、找回密码流程可用；重置后旧 Token 失效
- [ ] `/health` 正常，Docker 一键部署文档可复现

## 14. 风险与约束

| 风险 | 说明 | 缓解 |
|------|------|------|
| 密钥丢失 | 无备份则历史永久丢失 | 引导用户完成链接或云备份 |
| 境外访问 | 国内用户可能延迟高或偶发阻断 | 选近区节点；必要时告知用户网络环境 |
| E2EE 复杂度 | 群 epoch、多设备同步实现难度大 | 分阶段交付，充分测试入群与换机 |
| 群主离线 | 入群 epoch 轮换依赖群主或指定设备 | 可指定 co-admin 或预置 encrypted epoch proposal |
